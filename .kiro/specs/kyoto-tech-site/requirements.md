# Requirements Document

## Introduction

This document captures the requirements for the Kyoto Tech Meetup website — an Astro 5 static site deployed on Cloudflare Pages at https://kyototechmeetup.com. The site serves as the primary marketing and community hub for the Kyoto Tech Meetup group. It presents upcoming events pulled live from Meetup.com at build time, aggregates RSS/Atom feeds from member blogs and YouTube channels, and distributes new community content to Discord and optional webhook destinations via a scheduled notifier.

The spec covers the full system: the public-facing single-page site (English + Japanese), build-time data pipelines, community feed aggregation and notification, image optimisation, CI/CD automation, the legacy GitHub Pages redirect, and the known fragilities that must be managed as ongoing risks.

---

## Glossary

- **Site**: The Astro static site served from https://kyototechmeetup.com (Cloudflare Pages).
- **Build**: The `astro build` process that produces the static `dist/` output, including live Meetup event fetching and feed fallback.
- **Meetup Scraper**: The `fetchMeetupEvents()` function in `src/lib/meetup-events.ts` that fetches https://www.meetup.com/kyoto-tech-meetup/events/, extracts the `__NEXT_DATA__` JSON blob, and parses upcoming events from the embedded Apollo cache.
- **Event Type**: One of three classifications applied to a Meetup event title — `coffee`, `hack-day`, or `special`.
- **Calendar**: The `FullCalendarView` React island (`src/components/FullCalendarView.jsx`, using `@fullcalendar/react`) that renders a five-week day-grid of upcoming events in JST timezone.
- **Composite Feed**: The `src/data/composite-feed.json` file generated by `scripts/fetch-feeds.mjs`, consumed at build time by the Community Feed section.
- **Feed Aggregator**: `scripts/fetch-feeds.mjs` — the Node script that fetches RSS/Atom feeds for each Member Feed source, normalises items, resolves images, and writes `composite-feed.json`.
- **Member Feed**: A single source entry in `src/data/member-feeds.json` with a stable `id`, `name`, `siteUrl`, and `feedUrl`.
- **Notifier**: `scripts/community-feed-notifier.mjs` — the Node script that compares current feed items against Gist-backed state and delivers new items to configured destinations.
- **Notifier State**: A JSON document persisted in GitHub Gist `f95dd7597eec170d738d905e3666bfc6` as `community-feed-state.json`, tracking which feed items have been seen and delivered and which Meetup event reminders have been sent.
- **Event Reminder**: A Discord notification sent at a configured time offset before a Meetup event's start time (e.g., 24 hours before, 1 hour before).
- **Weekly Digest**: A Discord notification sent every Monday at 08:00 JST summarising all upcoming Meetup events for the week ahead (Monday–Sunday).
- **Reminder Window**: A named time-offset label (`24h`, `1h`) identifying a specific reminder type for a given event; together with the event ID these form the composite key used to deduplicate reminder deliveries in Notifier State.
- **Meetup Event Reminder Script**: `scripts/meetup-event-reminder.mjs` — the Node script that fetches upcoming Meetup events, determines which reminders and weekly digests are due, and delivers them to configured destinations.
- **Gist Token**: `GH_GIST_TOKEN` — a GitHub Personal Access Token with the `gist` scope, stored as a GitHub Actions secret.
- **Layout**: `src/layouts/Layout.astro` — the shared HTML shell, head meta, structured data, GTM injection, hreflang, canonical URL, and client-side language redirect logic.
- **LanguagePicker**: `src/components/LanguagePicker.astro` — the UI control that writes the user's language preference to `localStorage` under the key `kyoto-tech-language`.
- **i18n**: The Astro-native internationalisation layer; English (`en`) is the default locale, Japanese (`ja`) is served from `/ja/`.
- **Image Optimiser**: `scripts/optimize-images.mjs` — the sharp-powered script that resizes and compresses images under `public/images/`, tracking state in `assets/raw-images/.optimized/` sentinel JSON files.
- **Cloudflare Deploy Hook**: `CLOUDFLARE_DEPLOY_HOOK` secret — a URL that triggers a Cloudflare Pages build when POSTed to.
- **GitHub Pages Redirect**: A minimal static site in `.github/redirect-site/` that redirects visitors from https://kyoto-tech.github.io/ to https://kyototechmeetup.com/.
- **GTM**: Google Tag Manager, ID `GTM-5SB4S7NJ`, injected in every page `<head>`.

## Requirements

### Requirement 1: Single-Page Site Structure

**User Story:** As a community visitor, I want to find all essential community information — events, feeds, why to join, who attends — on one cohesive page, so that I can evaluate the group without navigating away.

#### Acceptance Criteria

1. THE Site SHALL render a single HTML page per locale containing the following named sections in order: Hero, Quick Links, Calendar (which contains the event legend and a Frequent Locations sub-section with anchor `id="locations"`), Community Feed, Why Join, Who Comes, and Footer.
2. THE Site SHALL expose anchor IDs `quick-links`, `calendar`, `locations`, `community-feed`, `why-join`, and `who-comes` so that the fixed Header navigation links scroll directly to the corresponding sections; after scrolling, each section heading SHALL be fully visible and not obscured by the fixed Header.
3. THE Layout SHALL include a skip-link pointing to `#main-content` so that keyboard and assistive-technology users can bypass repeated navigation.
4. THE Layout SHALL inject the GTM container script (`GTM-5SB4S7NJ`) in the `<head>` of every page and the corresponding `<noscript>` iframe at the top of `<body>`.
5. THE Layout SHALL use Astro View Transitions (`ClientRouter` from `astro:transitions`) so that navigating between locale pages updates page content and the URL without a full browser reload.
6. THE page structure SHALL be implemented via a root `src/pages/index.astro` that accepts a `lang` prop (defaulting to `"en"`); the Japanese page at `src/pages/ja/index.astro` SHALL re-use the root page component by passing `lang="ja"` rather than duplicating page markup.

### Requirement 2: Bilingual i18n (English and Japanese)

**User Story:** As a Japanese-speaking visitor, I want the site to present content in Japanese when my browser preference or stored preference indicates Japanese, so that I can engage with the community in my preferred language.

#### Acceptance Criteria

1. THE Site SHALL serve English content from the root path (`/`) and Japanese content from `/ja/`, using Astro's native i18n routing with `en` as the default locale.
2. WHEN a visitor loads the home page at either locale path (`/` or `/ja/`), THE Layout SHALL run an inline script before first paint that reads `localStorage.getItem("kyoto-tech-language")` and, if that value is absent or neither `"en"` nor `"ja"`, falls back to `navigator.language` / `navigator.languages` to detect a Japanese browser preference.
3. WHEN path comparison is performed in the language-redirect script, THE Layout SHALL normalise both the current path and the locale home paths by stripping any trailing `index.html` suffix and ensuring a trailing slash, so that the redirect fires correctly regardless of how the path is represented.
4. WHEN the detected target language does not match the current locale URL, THE Layout SHALL call `location.replace()` to redirect to the target locale home path (`/` for English, `/ja/` for Japanese) before the page renders visually.
5. WHEN a visitor selects a language using the LanguagePicker, THE LanguagePicker SHALL write the chosen locale value (`"en"` or `"ja"`) to `localStorage` under the key `kyoto-tech-language`; IF `localStorage` is unavailable, THE LanguagePicker SHALL fail silently without throwing.
6. THE Site SHALL include `<link rel="alternate" hreflang="en" href="https://kyototechmeetup.com/">`, `<link rel="alternate" hreflang="ja" href="https://kyototechmeetup.com/ja/">`, and `<link rel="alternate" hreflang="x-default" href="https://kyototechmeetup.com/">` tags in every page `<head>`.
7. THE Site SHALL resolve all user-visible strings through the `useTranslations(lang)` helper, which falls back to the `en` string table when a key is absent in the `ja` table.
8. WHEN a string key is present in `en` but absent or empty in `ja`, THE `useTranslations` helper SHALL return the English value rather than an empty string or an error.

### Requirement 3: SEO and Structured Data

**User Story:** As a search engine crawler, I want each locale page to expose correct canonical URLs, Open Graph metadata, Twitter card metadata, and Schema.org structured data, so that the site is indexed and previewed correctly.

#### Acceptance Criteria

1. THE Layout SHALL emit a `<link rel="canonical">` tag whose `href` resolves to the locale-specific home URL (e.g., `https://kyototechmeetup.com/` for English and `https://kyototechmeetup.com/ja/` for Japanese).
2. THE Layout SHALL set `og:locale` to `en_US` on English pages and `ja_JP` on Japanese pages, and SHALL set `og:locale:alternate` to the opposite value.
3. THE Layout SHALL include a `<script type="application/ld+json">` block containing a Schema.org `@graph` with: an `Organization` node (with `founder` and `organizer` references, `sameAs` links to Meetup and GitHub, and `areaServed` set to Kyoto), a `Person` node for the founder (with `name`, `url`, `sameAs`, and `jobTitle`), a `WebSite` node, two `EventSeries` nodes (Morning Tech & Coffee and Community Hack Day), and a `WebPage` node whose `url` equals the canonical URL for the current locale and whose `inLanguage` equals the active locale code (`"en"` or `"ja"`).
4. THE Layout SHALL set the HTML `lang` attribute to `activeLang` (`"en"` or `"ja"`) on the `<html>` element.
5. THE Site SHALL generate a sitemap via `@astrojs/sitemap` using `https://kyototechmeetup.com` as the site URL; the sitemap SHALL include both `https://kyototechmeetup.com/` and `https://kyototechmeetup.com/ja/`.
6. THE Layout SHALL include `<meta name="twitter:card">`, `<meta name="twitter:title">`, `<meta name="twitter:description">`, `<meta name="twitter:image">`, and `<meta name="twitter:image:alt">` tags on every page, with values matching the corresponding Open Graph tags.
7. THE Layout SHALL include favicon and web manifest link tags referencing assets under `/kyoto-tech-favicon/`, covering the standard set of browser and OS icon formats (ICO, SVG, PNG, Apple Touch Icon, and `site.webmanifest`); specific sizes are determined by the generated favicon asset set and should not be hardcoded in this requirement.

### Requirement 4: Meetup Event Fetching (Build-Time Scraper)

**User Story:** As a site visitor, I want to see accurate upcoming Kyoto Tech events on the calendar, so that I can plan my attendance.

#### Acceptance Criteria

1. WHEN the Build runs, THE Meetup Scraper SHALL perform an HTTP GET to `https://www.meetup.com/kyoto-tech-meetup/events/` with a cache-busting query parameter and `cache-control: no-cache` headers.
2. WHEN the HTTP response contains a `<script id="__NEXT_DATA__">` element, THE Meetup Scraper SHALL parse the embedded JSON and extract the `props.pageProps.__APOLLO_STATE__` map.
3. WHEN `__APOLLO_STATE__` contains entries whose keys start with `Event:` and whose `dateTime` field is non-null, THE Meetup Scraper SHALL map each entry to a `MeetupEvent` object with fields: `title`, `link`, `start`, `endTime`, `description`, `image`, `goingCount`, `interestedCount`, `eventType`, and `venue`; optional fields (`description`, `image`, `venue`) SHALL be `null` when absent in the source data.
4. THE Meetup Scraper SHALL include in the returned list only events whose start time is within the next 60 days from build time OR that are currently in progress (end time not yet passed, or — when end time is absent — start time was within the last 4 hours).
5. THE Meetup Scraper SHALL strip the suffix `" | Kyoto Tech Meetup"` from event titles and, for events classified as `coffee`, SHALL additionally strip a trailing `" on {Weekday}"` suffix, where `{Weekday}` is any English day name (Monday through Sunday), case-insensitively.
6. THE Meetup Scraper SHALL classify each event by applying rules in priority order — `coffee` first (title contains "coffee", case-insensitive), then `hack-day` (title matches `\bhack[-\s]+day\b`, case-insensitive), then `special` for all other titles — so that a title matching both patterns is always classified as `coffee`.
7. IF the HTTP response returns a non-200 status code, does not contain a `<script id="__NEXT_DATA__">` element, contains a `__NEXT_DATA__` element with malformed JSON, OR if the network request itself throws (e.g. DNS failure, connection timeout), THEN THE Meetup Scraper SHALL catch the error and return an empty array rather than allow the exception to propagate to the build.
8. THE Meetup Scraper SHALL sort returned events by ascending start time.

> **Known Fragility — Meetup Scraper:** The `__NEXT_DATA__` extraction depends on Meetup.com's undocumented internal data format. Any change to Meetup's frontend rendering pipeline (Next.js upgrade, SSR refactor, Apollo cache restructuring) may silently break event fetching. The site degrades gracefully to an empty calendar when the scraper returns no events; however, there is no build-time alerting when events are unexpectedly absent.

### Requirement 5: Event Calendar Display

**User Story:** As a visitor, I want to browse upcoming events in a visual month-style calendar so that I can quickly identify event dates and types.

#### Acceptance Criteria

1. THE Calendar SHALL render a five-week grid starting from the Monday of the current week and spanning exactly 35 days, displaying all events in Japan Standard Time (JST, `Asia/Tokyo` timezone) regardless of the visitor's local timezone.
2. THE Calendar SHALL render each event cell with a distinct visual style per Event Type (`coffee`, `hack-day`, `special`) so that each type is visually distinguishable from the others.
3. WHEN an event cell is clicked, THE Calendar SHALL open the event's Meetup.com URL in a new browser tab.
4. WHEN the Meetup Scraper returns no events or returns an error, THE Calendar Section SHALL display a fallback message with a link to the Meetup group page.
5. THE Calendar section SHALL include a legend showing one card per Event Type, each with a representative photo thumbnail, a colour indicator matching that event type's calendar colour, a title, and a description of no more than 20 words.
6. THE Calendar section SHALL include a Frequent Locations sub-section rendering one card per entry in `src/data/event-locations.json`, each containing a venue `name` and an `embedUrl` field; each entry SHALL be rendered as a Google Maps iframe whose `title` attribute uses the `a11y.mapTitle` i18n key and whose `aria-label` attribute uses the `a11y.mapLabel` i18n key, both interpolated with the venue name.
7. THE Calendar section SHALL render the same `FeedTimestamps` client-side island (Req 7.5) immediately after the calendar grid, passing the same `generatedAt`, interval, and localised labels, so that visitors can see when event data was last fetched and when it will next refresh.

### Requirement 6: Community Feed Aggregation (Build-Time)

**User Story:** As a site visitor, I want to read the latest posts from community members' blogs and YouTube channels so that I can stay informed about what members are building and thinking about.

#### Acceptance Criteria

1. WHEN the Build runs, THE Feed Aggregator SHALL read `src/data/member-feeds.json` and fetch the RSS/Atom feed for each Member Feed source within a configurable timeout (default 12 seconds, configurable via `COMPOSITE_FEED_TIMEOUT_MS`); image resolution fetches from item-linked pages SHALL use a separate timeout (default 8 seconds).
2. WHEN a Member Feed `feedUrl` matches a YouTube `@handle` URL, THE Feed Aggregator SHALL scrape the channel page to extract the numeric channel ID from the page HTML before fetching the RSS feed URL `https://www.youtube.com/feeds/videos.xml?channel_id={id}`.
3. WHEN items are fetched for a Member Feed, THE Feed Aggregator SHALL normalise each item to include: `id`, `title`, `link`, `publishedAt` (ISO 8601), `summary` (truncated to 360 characters), `image`, and `source` fields; the `source` field SHALL be an object containing the member feed's `name`, `siteUrl`, and `feedUrl`.
4. THE Feed Aggregator SHALL attempt to resolve a representative image for each item using, in priority order: enclosure URL, media thumbnail/content, inline `<img>` in HTML content, and Open Graph / Twitter meta tags fetched from the item's linked page.
5. IF a YouTube channel feed is being processed AND no higher-priority image source is found, THEN THE Feed Aggregator SHALL derive the thumbnail URL as `https://i.ytimg.com/vi/{videoId}/hqdefault.jpg`.
6. THE Feed Aggregator SHALL deduplicate items within a single feed by `id`, return at most `itemsPerFeed` items per source (minimum 1, default 3, configurable via `--items-per-feed` or `COMPOSITE_FEED_ITEMS_PER_FEED`), sorted by descending publish date.
7. THE Feed Aggregator SHALL write the output to `src/data/composite-feed.json` with: a top-level `generatedAt` ISO timestamp, `itemsPerFeed` count, `feeds` array (one entry per Member Feed with `id`, `name`, `siteUrl`, `items`, and optional `error`), and `failedSources` array (one entry per failed source with `id` and `error` message).
8. WHEN all feeds return zero items AND `--stale-ok` is passed AND the existing `composite-feed.json` is readable, THE Feed Aggregator SHALL fall back to the existing content (updating only `generatedAt` and setting `usedFallback: true`) rather than overwrite with empty data; IF the existing file is absent or unreadable, THE Feed Aggregator SHALL write an empty result regardless of `--stale-ok`.
9. IF fetching a Member Feed fails, THEN THE Feed Aggregator SHALL record the error in `failedSources`, include an `error` field on the affected feed entry in `feeds`, log a warning, and continue processing remaining sources.
10. THE Build script (`npm run build`) SHALL invoke `npm run feeds:pull:stale-ok` (i.e., Feed Aggregator with `--stale-ok`) before running `astro build` so that the Community Feed section always has data even when feed fetching fails transiently.

> **Known Fragility — YouTube Feed Resolution:** Resolving a YouTube `@handle` URL to a channel ID requires scraping the YouTube channel page for the `externalId` / `channelId` value embedded in the page HTML. YouTube may change its page structure, rendering the channel ID extraction unreliable.

### Requirement 7: Community Feed Display

**User Story:** As a visitor, I want to see recent posts from each member's feed displayed in a scannable card layout so that I can find interesting content to read.

#### Acceptance Criteria

1. THE Community Feed Section SHALL render one article card per Member Feed, containing the feed name, a link to the feed's `siteUrl`, and a horizontally scrollable row of post cards.
2. THE Community Feed Section SHALL display for each post card: the publication date formatted in JST, the post title as an external link, an image (float-right, shown only when the item's `image` field is a non-null non-empty string), an optional summary excerpt, and a "Read post" link using the `home.communityFeed.readMore` i18n key.
3. WHEN a feed has no items, THE Community Feed Section SHALL display the `home.communityFeed.emptyFeed` localised string in place of the post card row.
4. WHEN a feed's `error` field is set in `composite-feed.json`, THE Community Feed Section SHALL display a badge using the `home.communityFeed.fetchError` i18n key on the feed article header.
5. IF `composite-feed.json` contains a valid ISO 8601 `generatedAt` timestamp, THEN THE Community Feed Section SHALL render a `FeedTimestamps` client-side React island that displays a "Last updated" row using the `home.communityFeed.updated` i18n key with the time formatted in the visitor's local OS timezone using `toLocaleTimeString(undefined, ...)`, and a "Next update" row using the `home.communityFeed.nextUpdate` i18n key with the time calculated as `generatedAt + 3 hours` also formatted in the visitor's local timezone; IF `generatedAt` is absent or unparseable, THEN the component SHALL render nothing. The "Last updated" row SHALL use a `FaCheckCircle` icon and the "Next update" row SHALL use a `FaArrowRight` icon.
6. THE Community Feed Section SHALL render all strings (section title, action labels, empty-state messages, update timestamps) using the `useTranslations(lang)` helper.
7. THE Community Feed Section SHALL match composite feed entries to member feeds by comparing `feedUrl`, `siteUrl`, or `name` fields so that feed metadata (name, error state) is correctly associated with the feed display even if source ordering changes.

### Requirement 8: Community Feed Notifier

**User Story:** As a community member subscribed to Discord or a generic webhook, I want to receive a notification when any member publishes a new post, so that I can read it promptly.

#### Acceptance Criteria

1. THE Notifier SHALL read Member Feed sources from `src/data/member-feeds.json` and fetch each feed within a configurable timeout (default 12 seconds, env `COMMUNITY_FEED_TIMEOUT_MS`).
2. THE Notifier SHALL fetch at most `COMMUNITY_FEED_MAX_ITEMS_PER_FEED` items per feed (default 10) when reading feed contents.
3. THE Notifier SHALL read current Notifier State from the GitHub Gist identified by `COMMUNITY_FEED_STATE_GIST_ID` (default `f95dd7597eec170d738d905e3666bfc6`) using a bearer token (`GH_GIST_TOKEN`).
4. WHEN the Notifier State is empty (first run) and `--allow-initial-posts` is not set, THE Notifier SHALL record all current feed items as suppressed in the Notifier State without delivering any notifications, then write the updated state back to the Gist.
5. WHEN the Notifier State is empty (first run) and `--allow-initial-posts` is set, THE Notifier SHALL process and attempt to deliver all current feed items as if they were new.
6. WHEN a feed item's `id` is not present in the Notifier State or has not been delivered to a destination, THE Notifier SHALL attempt delivery to each configured destination (Discord webhook and/or generic webhook).
7. WHEN `DISCORD_WEBHOOK_URL` is set, THE Notifier SHALL POST a Discord embed payload to the webhook URL for each new item; the embed SHALL include title, URL, timestamp, author name, author URL, description (summary if present), and footer.
8. WHEN `COMMUNITY_FEED_GENERIC_WEBHOOK_URL` is set, THE Notifier SHALL POST a JSON payload `{ event: "community_feed_item", item, message }` to the webhook URL for each new item.
9. WHEN neither `DISCORD_WEBHOOK_URL` nor `COMMUNITY_FEED_GENERIC_WEBHOOK_URL` is set and `--skip-without-destinations` is passed, THE Notifier SHALL exit without error rather than throw.
10. WHEN neither `DISCORD_WEBHOOK_URL` nor `COMMUNITY_FEED_GENERIC_WEBHOOK_URL` is set and `--skip-without-destinations` is NOT passed, THE Notifier SHALL exit with a non-zero exit code and log an error indicating no destinations are configured.
11. WHEN delivery to a destination succeeds, THE Notifier SHALL record `deliveredAt`, `deliveryId` (may be `null` for destinations that return no message ID), and `lastAttemptAt` on the state record's `channels` entry for that destination and write state to the Gist.
12. WHEN delivery to a destination fails, THE Notifier SHALL record the error in `lastError` on the state record's `channels` entry, and after processing all items SHALL exit with a non-zero exit code.
13. IF fetching a single Member Feed fails, THEN THE Notifier SHALL log a warning and continue processing remaining feeds; partial fetch failure SHALL NOT cause a non-zero exit code unless all feeds fail.
14. WHEN `--dry-run` is passed, THE Notifier SHALL log what would be sent without posting to any destination or updating Gist state.
15. WHEN `--max-deliveries N` and `--suppress-remaining-after-limit` are passed, THE Notifier SHALL deliver at most N items and mark additional pending items as suppressed in the Notifier State; this is the mechanism used by the `demo_mode` workflow input in Req 12.2, which passes `--max-deliveries 3 --suppress-remaining-after-limit`.
16. THE Notifier SHALL migrate Notifier State item IDs from the legacy `feedUrl::itemId` format to the current `feedSourceId::itemId` format when a version mismatch is detected, writing the migrated state back to the Gist before processing deliveries.

> **Known Fragility — Gist-Backed State:** Notifier State is stored in a public GitHub Gist with no distributed locking. If two Notifier workflow runs overlap (possible if a scheduled run is delayed into the next 15-minute window), both runs may read the same base state and each write back a divergent state, potentially causing duplicate deliveries or lost delivery records. The `concurrency: cancel-in-progress: false` setting in the workflow prevents cancellation but does not prevent races if a previous run is still executing when the next is queued.

### Requirement 9: Image Optimisation Pipeline

**User Story:** As a contributor adding new images to the site, I want a repeatable optimisation step that resizes and compresses images so that page load performance is not degraded by oversized assets.

#### Acceptance Criteria

1. THE Image Optimiser SHALL process all image files (`.png`, `.jpg`, `.jpeg`, `.webp`, `.avif`, `.gif`) found recursively under `public/images/`; for JPEG output, the file extension SHALL be normalised to `.jpg` (so `photo.jpeg`, `photo.webp`, `photo.avif`, and `photo.gif` are all written as `photo.jpg`, replacing the original file in place); PNG files retain their `.png` extension.
2. WHEN processing an image, THE Image Optimiser SHALL resize it to a maximum width of 1600 pixels (preserving aspect ratio, without enlargement), output JPEG at 85% quality (mozjpeg), and output PNG at compression level 9 with adaptive filtering.
3. IF no raw backup file exists at `assets/raw-images/` for an image, THEN THE Image Optimiser SHALL copy the original file to `assets/raw-images/` as a backup before overwriting the public copy.
4. IF a sentinel JSON file exists in `assets/raw-images/.optimized/` for an image AND a raw backup file is present, THEN THE Image Optimiser SHALL skip the image and log that it is already optimised.
5. IF `--use-raw` is passed, THEN THE Image Optimiser SHALL reprocess the image from the raw backup even when the sentinel file is present.
6. WHEN `--check` mode is run, THE Image Optimiser SHALL perform a dry-run assessment and exit with a non-zero exit code if any images would be optimised, enabling CI enforcement without modifying files.
7. IF `public/images/` does not exist, THEN THE Image Optimiser SHALL exit with a non-zero exit code and log an appropriate error.
8. THE `npm run images:check` command SHALL invoke the Image Optimiser with `--check`, and the `npm run check` script SHALL include `images:check` so that CI fails when unoptimised images are committed.

### Requirement 10: CI/CD Pull Request Build Checks

**User Story:** As a contributor opening a pull request, I want automated checks to verify that linting, type checking, Astro diagnostics, and the production build all pass before merging, so that regressions are caught early.

#### Acceptance Criteria

1. WHEN a pull request is opened or updated against the repository, THE `build.yml` workflow SHALL run `npm ci` followed by the steps defined in the `npm run check` script (Req 15.4: lint, test, tsc, astrocheck, images:check, knip) and then `npm run build`; the `npm run check` script is the single authoritative definition of the quality gate step sequence.
2. THE `build.yml` workflow SHALL use Node.js 20 with npm cache, run on `ubuntu-latest`, and complete within 15 minutes.
3. THE `build.yml` workflow SHALL use `concurrency: cancel-in-progress: true` scoped to the PR number so that a new push cancels any in-progress run for the same PR.
4. IF any step in the `build.yml` workflow fails, THEN the workflow job SHALL exit with a non-zero status, blocking merge.

### Requirement 11: CI/CD Scheduled Site Rebuild

**User Story:** As a community member visiting the site, I want event data and the community feed to refresh automatically every few hours so that the content stays current without manual deploys.

#### Acceptance Criteria

1. THE `scheduled-build.yml` workflow SHALL trigger a Cloudflare Pages rebuild by POSTing to the `CLOUDFLARE_DEPLOY_HOOK` URL on a cron schedule of every 3 hours (`0 */3 * * *`).
2. THE `scheduled-build.yml` workflow SHALL also be triggerable manually via `workflow_dispatch`.
3. IF `CLOUDFLARE_DEPLOY_HOOK` is not set in repository secrets, THEN THE `scheduled-build.yml` workflow SHALL exit with a non-zero status and log an error rather than silently skip.
4. THE `scheduled-build.yml` workflow SHALL use `concurrency: cancel-in-progress: true` so that a new cron trigger or manual dispatch supersedes any in-progress run.

### Requirement 12: CI/CD Community Feed Notifier Workflow

**User Story:** As a Discord channel subscriber, I want new member blog posts and YouTube videos to be announced in the channel within 15 minutes of being published, so that I can read them promptly.

#### Acceptance Criteria

1. THE `community-feed-notifier.yml` workflow SHALL run the Notifier on a cron schedule of every 15 minutes (`*/15 * * * *`).
2. THE `community-feed-notifier.yml` workflow SHALL support manual `workflow_dispatch` with three boolean inputs: `demo_mode` (post at most 3 items, suppress rest; default `false`), `dry_run` (check without posting or updating state; default `true`), and `allow_initial_posts` (post backlog on first run; default `false`).
3. THE `community-feed-notifier.yml` workflow SHALL pass `--skip-without-destinations` to the Notifier when neither `DISCORD_WEBHOOK_URL` nor `COMMUNITY_FEED_GENERIC_WEBHOOK_URL` is configured, allowing scheduled runs to exit cleanly.
4. THE `community-feed-notifier.yml` workflow SHALL set `COMMUNITY_FEED_STATE_GIST_ID` to `f95dd7597eec170d738d905e3666bfc6` and `COMMUNITY_FEED_MAX_ITEMS_PER_FEED` to `10` as environment variables for the Notifier.
5. THE `community-feed-notifier.yml` workflow SHALL use `concurrency: cancel-in-progress: false` so that a queued run is never cancelled by a later trigger.
6. THE `community-feed-notifier.yml` workflow job SHALL have a `timeout-minutes: 10` limit.

### Requirement 13: CI/CD GitHub Pages Redirect

**User Story:** As a visitor arriving at the legacy `https://kyoto-tech.github.io/` URL, I want to be transparently redirected to `https://kyototechmeetup.com/` so that bookmarks and external links continue to work.

#### Acceptance Criteria

1. ### Requirement 17: Meetup Event Reminder Notifier

**User Story:** As a Discord channel subscriber, I want to receive a reminder when a Kyoto Tech event is approaching and a digest of the week's events every Monday morning, so that I can plan my attendance without manually checking Meetup.

#### Acceptance Criteria

1. THE Meetup Event Reminder Script (`scripts/meetup-event-reminder.mjs`) SHALL be a standalone Node script that accepts the same `--dry-run` and `--skip-without-destinations` CLI flags as the community feed notifier and shares the same Gist state read/write utilities from `scripts/lib/community-feed-notifier-state.mjs` and HTTP utilities from `scripts/lib/community-feed-reader.mjs`.
2. WHEN run, THE Meetup Event Reminder Script SHALL fetch upcoming events using the existing `fetchMeetupEvents()` function from `src/lib/meetup-events.ts`; IF the scraper returns an empty array or throws, THE script SHALL log a warning and exit cleanly with a zero exit code (treating no events as a non-error condition).
3. WHEN `COMMUNITY_FEED_STATE_GIST_ID` is set, THE Meetup Event Reminder Script SHALL read and write Notifier State from the GitHub Gist identified by that environment variable using the bearer token in `GH_GIST_TOKEN`; IF `COMMUNITY_FEED_STATE_GIST_ID` is absent and `--dry-run` is not set, THE script SHALL exit with a non-zero exit code and log an error.
4. THE Notifier State schema SHALL be extended to version 3 by adding a top-level `events` map alongside the existing `items` map; keys in `events` SHALL be the Meetup event `link` URL (used as a stable event identifier); WHEN an event `link` is encountered that is not yet present in `state.events`, THE script SHALL initialise a new entry with `eventId` (the link URL), `title`, `start` (ISO 8601), `reminders` set to an empty object, and `weeklyDigest` set to an empty object before evaluating delivery; each `reminders` entry is keyed by Reminder Window label (`"24h"` or `"1h"`) and records `deliveredAt` (ISO 8601 or `null`), `deliveryId` (string or `null`), `lastAttemptAt` (ISO 8601 or `null`), and `lastError` (string or `null`); the top-level `weeklyDigest` object on the state (not on individual event entries) is keyed by ISO week identifier (e.g., `"2026-W25"`) and each value records the same four delivery fields; `deliveryId` for a weekly digest entry SHALL be the Discord message ID returned from the single weekly digest POST, or `null` if Discord does not return one.
5. THE `migrateStateItemIds` function SHALL be updated to handle version 2→3 migration: it SHALL initialise `state.events` as an empty object when absent and initialise the top-level `state.weeklyDigest` as an empty object when absent, then bump `state.version` to 3, preserving all existing `state.items` data unchanged.
6. WHEN the current wall-clock time (UTC) is at or past `event.start − 24 hours` and the `"24h"` reminder for that event has not yet been delivered (`deliveredAt` is `null`), THE script SHALL send a pre-event reminder to Discord; WHEN the current wall-clock time is at or past `event.start − 1 hour` and the `"1h"` reminder has not yet been delivered, THE script SHALL send a pre-event reminder to Discord; WHEN `DISCORD_WEBHOOK_URL` is set, THE script SHALL POST a Discord embed payload with `?wait=true`; the `content` field SHALL read `⏰ Event reminder — **{eventType emoji}** {event title}`; the embed SHALL include: `title` (event title), `url` (Meetup event link), `timestamp` (ISO 8601 event start), `description` formatted as `📍 {venueName}\n👥 {N} going · {M} interested` (venue line omitted entirely when venue is absent), and `footer.text` set to `Kyoto Tech Meetup`; the event type emoji mapping SHALL be: `coffee` → ☕, `hack-day` → 💻, `special` → ⭐.
7. WHEN the current day is Monday in JST (`Asia/Tokyo` timezone) AND the current time is at or past 08:00 JST AND a weekly digest for the current ISO week identifier has not yet been delivered (`deliveredAt` is `null`), THE script SHALL send a weekly digest to Discord; WHEN `DISCORD_WEBHOOK_URL` is set, THE script SHALL POST a single Discord message with `content` set to `📅 This week's Kyoto Tech events:` and a single embed whose `description` field is a newline-separated list of bullet lines, one per event occurring in the current Monday–Sunday week (boundaries evaluated in JST), each formatted as `• [{event title}]({meetup link}) — {ddd D MMM YYYY [at] HH:mm [JST]}[ · 📍 {venueName}]` (venue suffix omitted when absent); IF no events fall in the current week, THE script SHALL NOT POST a weekly digest message and SHALL NOT create a state entry for that ISO week.
8. WHEN delivery to Discord succeeds, THE script SHALL record `deliveredAt` (ISO 8601 timestamp of the attempt), `deliveryId` (Discord message ID or `null`), and `lastAttemptAt` on the corresponding `reminders[window]` or `state.weeklyDigest[isoWeek]` entry and write the updated state back to the Gist immediately after each delivery.
9. WHEN delivery to Discord fails, THE script SHALL record `lastError` (the error message string), `deliveredAt: null`, and `lastAttemptAt` on the state entry, write the updated state to the Gist immediately, continue processing any remaining due reminders, and after all reminders have been evaluated SHALL exit with a non-zero exit code.
10. WHEN `--dry-run` is passed, THE script SHALL log what would be sent (including the reminder window, event title, and target) without posting to Discord or updating Gist state.
11. WHEN `DISCORD_WEBHOOK_URL` is not set and `--skip-without-destinations` is passed, THE script SHALL exit cleanly with a zero exit code and log that no destinations are configured; WHEN `--skip-without-destinations` is not passed and `DISCORD_WEBHOOK_URL` is not set, THE script SHALL exit with a non-zero exit code and log an error.
12. THE Meetup Event Reminder Script SHALL write state back to the Gist immediately after each individual delivery attempt (success or failure), not in a single batch at the end, so that partial progress is preserved if the script is interrupted mid-run.

> **Known Fragility — Event Reminder Scheduling:** Because the reminder script runs on a cron schedule (Req 18) rather than at exact offset times, there is an inherent window of up to one cron interval (15 minutes) between when a reminder becomes due and when it fires. Reminders MUST be considered due if the current time is at or past the trigger time, not exactly equal, so that a slightly delayed cron run still delivers the reminder.

### Requirement 16: Known Limitations and Future Direction

**User Story:** As a maintainer, I want known limitations and deferred work to be documented so that contributors understand the risks and the intended growth path.

#### Acceptance Criteria

1. THE Site SHALL be documented as a single-page architecture per locale, and any decision to expand to multi-page routing SHALL require a design review given the current navigation model (anchor-based scroll navigation, no dedicated event detail or blog pages).
2. The `@astrojs/mdx` integration is installed and configured in `astro.config.mjs` but currently unused by any pages. It is available infrastructure for future MDX-based content pages; whether to retain or remove it should be decided when Knip dead-code enforcement is evaluated against the integration configuration.
3. THE Meetup Scraper fragility (Requirement 4, `__NEXT_DATA__` dependence) SHALL be treated as an active operational risk; any future work to replace the scraper with an official Meetup API or a stable alternative SHALL be tracked as a first-class improvement. A related gap is that there is currently no build-time alerting when the scraper returns zero events unexpectedly; adding a warning or build failure when the event list is empty during a scheduled rebuild is a tracked future improvement.
4. THE Gist-backed Notifier State fragility (Requirement 8, no distributed locking) SHALL be treated as an active operational risk; future work to replace Gist state with a database or object-store backend with atomic writes SHALL be tracked as a first-class improvement.
5. THE YouTube feed resolution fragility (Requirement 6, page-scrape for channel ID) SHALL be treated as an active operational risk; adopting the YouTube Data API for channel resolution is the preferred mitigation path.
6. There is currently no documented process for adding a new venue to `src/data/event-locations.json` or a new member feed to `src/data/member-feeds.json`; contributor documentation (e.g., in `README.md`) covering how to add venues and feed sources is a tracked future improvement.
7. There is currently no performance budget defined for the site; setting targets for build time, Lighthouse scores, or Core Web Vitals is deferred and should be revisited when the site's traffic and page complexity grow.
